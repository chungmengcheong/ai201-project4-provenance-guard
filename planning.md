# Planning for Provenance Guard

## Purpose

Provenance guard classifies submitted text content, e.g,. a poem, a short story excerpt, a blog post, to whether it is generated by an AI, human, or uncertain. The detection pipeline combines two different  detection signals to generate a confidence score of whether it is AI generated. The user is shown a user-friendly label and can appeal the classification. 

## Architecture

The architecture revolves around two main flows:
- A submission flow that starts from a web-based UI that accepts user submission and initiates the detection flow. The result is displayed to the user in the same web UI. 
- An appeals flow that lets the users contest a classification within a web UI. The user's reasoning and the content status, which is set to "under review", is captured into the audit log.

## Submission flow

### Purpose
Web-based UI that accepts user submission and initiates the detection flow (`POST /submit`). The result is displayed to the user in the same web UI. 

### Input / Output Contract

**Input:**

content (str): user submitted text
creator_id (str): identifier of the submitting user

**Output:**

JSON:
{
'content_id' (str) : unique identifier for this submission (UUID),
'status' (str) : one of ['scored', 'error'],
'label' (str) : transparency label text,
'user_friendly_description' (str) : explanation of transparency label,
'confidence_score' (float) : range 0 to 1.0 inclusive, 
'signals' (dict) : {
    'LLM' (float) : range 0 to 1.0 inclusive,
    'LLM_reasoning' (str) : LLM reasoning text,
    'stylometric (float) : range 0 to 1.0 inclusive,
    }
'message' (str) : if error, otherwise default to Null
}    

### Design decisions

Logic flow:
1. The user submits text through the web interface

2. Check that user submitted text is between `MINIMUM_LENGTH` and `MAXIMUM_LENGTH`, inclusive. 

If not:
- Set 'label', 'user_friendly_description', 'confidence_score', 'signals'
- If text is less than `MINIMUM_LENGTH`, set status = 'error' and 'message' = 'Submitted text is too short to be assessed'.  
- If text is more than `MAXIMUM_LENGTH`, set status = 'error' and 'message' = 'Submitted text is too long to be assessed'.  
- Skip to step 6

3. Generate a unique `content_id` (UUID) for this submission.

4. Call detection pipeline with the user submitted text, which returns a confidence score along with other fields in a structured response.

5. Call apply_label() witht the confidence score, which returns a transparency label and an user-friendly description 

6. The decision/error is logged and returned as JSON. 

7. if status = 'scored', 'label' and 'user_friendly explanation' is shown to the user, along with the following message that allows them to enter the /appeal flow:

    "If you would like to appeal this decision, please enter your reasoning below and then click the 'Appeal' button"

8. if status = 'error', `message` is shown to the user.  

## Detection pipeline: detect_signal()

### Purpose
API endpoint that accepts a piece of text-based content (e.g., a poem, a short story excerpt, a blog post) for attribution analysis. The endpoint returns a confidence score and transparency label text as part of a structured response. The confidence ranges from 0 to 1.0 inclusive. A higher score means the user text is more likely to be generated by AI. 

### Input / Output Contract

**Input:**

content: str

**Output:**

dict:
{
'status' (str) : one of ['scored', 'error'],
'confidence_score' (float) : range 0 to 1.0 inclusive, 
'signals' (dict) : {
    'LLM' (float) : range 0 to 1.0 inclusive,
    'LLM_reasoning' (str) : LLM reasoning text,
    'stylometric (float) : range 0 to 1.0 inclusive,
    }
'message' (str) : if error, otherwise default to None
}    

### Design decisions

Logic flow:
1. The user submitted text is sent to two independent detection signal sub-systems. 
2. The aggregate confidence score is calcualted as simple average of the independent sub-systems calls, i.e, ((LLM score + sytlometric score) / 2)

#### Detection signal #1: LLM classification

### Purpose

The LLM is prompted to assess the likelihood of the content being generated by AI. 

LLMs can be fine-tuned with style guides that shows common AI stylistic tendencies (e.g., contrastive rhetorical framing such as it's not X, it's Y patterns). However, formally-written or carefully-edited human text (e.g., academic prose, business/legal writing, or non-native English speaker who writes in a structured style) may resemble AI writing and be incorrectly classified.

##### Input / Output Contract

**Inputs:**

content (str): of user provided content

**Output:**

Dict:
{
    confidence_score (float): range 0.0 to 1.0 inclusive,
    reasoning (str): text of LLM's reasoning,
    message (str): if error, otherwise default to None
}

##### Design decisions

Logic flow:
1. The LLM is provided with the following prompt:

```
Classify the following **content** on the likelihood that is written by AI. 

Respond in JSON:
{
    score_map: one of ["clearly_ai", "likely_ai", "uncertain", "likely_human", "clearly_human"],
    reasoning: 1 to 2 lines of text explaining your assessment
}

Writing that shows writing with several of and/or repeated use of the following stylistic tendencies are more likely to be AI writing:
- Contrastive Rhetorical Framing, e.g. "This isn't just about revenue diversification. It's about survival."
- Asking and Answering Rhetorical Questions, e.g., "“What changed? The math did.” [Human: The math changed.]
- Excessive use of dashes, bullets, and headings
- Triplet Framing, e.g., “Fast, cheap, and out of control.” Clever alliteration to a bosa nova beat.
- The Inspirational Pivot, e.g.: “This isn’t just about AI. It’s about humanity.” 
- Universal Authority Without Source, e.g., “Studies show that storytelling is 22 times more memorable than facts.”
- Quotes with incorrect attribution, e.g, "'AI is the new electricity,' said Musk."

##Content
{Content}
```

2. The output of the LLM is mapped to a numerical value (`confidence_map_to_score`):
  SCORE_MAP = {
      "clearly_ai":    0.95,
      "likely_ai":     0.75,
      "uncertain":     0.50,
      "likely_human":  0.25,
      "clearly_human": 0.05,
  }

3. If the LLM successfully returns a parseable result, set message = 'None'. Otherwise set status = 'error' and message = 'Error: User content was not parsed'

#### Detection signal #2: Stylometric heuristics

Stylometric heuristic is used to evaluate the likelihood of the content being generated by AI. 

Stylometric heuristic calculates the variability in text. Human writing tends to have more variability than AI. However this approach could misassess human writing that uses simple, repetitive vocabulary such as poetry or children stories as AI writing. 

##### Input / Output Contract

**Inputs:**

content (str): of user provided content

**Output:**

confidence_score (float): between 0.0 to 1.0 inclusive

##### Design decisions

AI text tends toward low sentence length variance (uniform rhythm) and more repetitive vocabulary. To detect this in text:

1. Compute the following:
- sentence_length_variance. higher = more human-like variability
- vocabulary_diversity. higher = more diverse vocabulary
2. Combine and normalize the above into an aggregated score of 0-1, where 1 is AI writing
- variance_score = 1 - min(sentence_length_variance / 50, 1)
- diversity_score = 1 - min(vocabulary_diversity / 0.95, 1)
- aggregate_score = (variance_score + diversity_score) / 2

Divisors calibrated against 4 test cases (clearly AI, clearly human, borderline formal human, borderline edited AI). Observed variance range was 24–45 across test cases; divisor 50 maps this to the full score range without capping any test case. Observed TTR range was 0.86–0.92; divisor 0.95 keeps all values within the active window. Note: for short texts, TTR has weak discriminative power and the variance signal carries most of the weight.

### Edge cases

Potential edge cases:
- Blank or very short content. Guard using a 'MINIMUM_LENGTH' requirement for user content. 
- Very long content. Guard using a 'MAXIMUM_LENGTH' requirement for user content. 
- User content is not English or gibberish (e.g., "ZDKJDFDFD"). Accept as known limitation for now and return label as "Unknown". 

## Apply label: apply_label()

### Purpose

Translate confidence score into a transparency label and a user friendly description

### Input / Output Contract

**Input:**

score (float): range 0 to 1.0 inclusive

**Output:**
dict: 
{
'label' (str) : transparency label text,
'user_friendly_description' (str) : explanation of transparency label
}

### Design decisions 

`score` is mapped to a label using the following table:

| Score | label | user friendly explanation |
|---|---|---|
| 0.90 - 1.0 | 'high-confidence AI' | "There is a high likelihood that the text is by an AI. We estimate that there is roughly 9 in 10 chance of this being AI generated." | 
| 0.26 - 0.89 | 'uncertain' |"It is difficult to say with high confidence whether the text is generated by an AI or written by a human."|
| 0 - 0.25 | 'high-confidence human' | "There is a strong likelihood that the text is by a human. We estimate that there is a roughly 3 in 4 chance of this being written by a human." |
 

## Appeals flow

### Purpose

Web-based UI that lets the users contest a classification. The user's reasoning and the content status, which is set to "under review", is captured into the audit log. 

### Input / Output Contract

**Inputs:**

dict:
{
'content_id' (str) : unique identifier from the preceding /submit response,
'creator_id' (str) : identifier of the submitting user,
'status' (str) : 'appeal',
'label' (str) : transparency label text,
'user_friendly_description' (str) : explanation of transparency label,
'confidence_score' (float) : range 0 to 1.0 inclusive, 
'signals' (dict) : {
    'LLM' (float) : range 0 to 1.0 inclusive,
    'LLM_reasoning' (str) : LLM reasoning text,
    'stylometric (float) : range 0 to 1.0 inclusive,
    }
'message' (str) : if error, otherwise default to Null
'user_reason' (str) : reason provided by user in the webUI 
}    

Note: 'label', 'user_friendly_description', 'confidence_score', 'signals', and 'message' are verbatim from the preceeding /endpoint response. 

**Output:**

'message' (str) : message to be displayed to user

#### Design decisions

Logic flow:
1. The user input is appended into the audit log 
2. If successfully logged, set 'message' to "Appeal successfully logged"
3. If not successfully logged, set 'message' to "There was an error with your submission. Please try again". 
4. The /appeal endpoint returns 'message', which is displayed by the webUI

## Production layer

### Purpose

This section documents the necessary mechanisms to enable the final production system:
- rate limiter
- logging

### Rate limiter

#### Purpose

The rate limiter is prevents users from abusing the detection pipeline by enforcing a timeout if serial submissions exceed the limit. 

#### Design decisions

Enforce a 'MAX_SUBMISSIONS_IN_TIME_WINDOW' (5) within a minute, and up to MAX_SUBMISSIONS_PER_DAY (100). This leverages Flask-limiter's window resets to simplify implementation. 

If 'MAX_SUBMISSIONS' is exceeded, the system will display the following error: 

    "There are too many submissions. Please try again later."

The rationale is that each submission is likely to take ~15s to go through the detection pipeline, which is ~4 requests per minutes. If the user is hitting the /submit endpoint more rapidly than 5 times in 1 minutes, or 100 times a day, they are more likely to be abusing the system.
